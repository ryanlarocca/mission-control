### 2. Rewrite AI summary format

**Current behavior:** `ai_summary` is generated by `analyzeCallTranscript` (and regenerated by Phase 7C's caching layer when new activity arrives). The current output is verbose and over-structured.

**New format Б─■ short paragraph with a temperature badge on top:**

```
П÷■╔ Hot
Brian called about a property he owns at 2127 Los Gatos Almaden Rd. He didn't share a timeline but sounded open to exploring options and wants a callback. Mentioned he's local to the area. Worth a quick follow-up tomorrow morning. Б≤▌О╦▐
```

Rules:
- Temperature badge first line: one of `П÷■╔ Hot` / `Б≤─О╦▐ Warm` / `Б²└О╦▐ Cold`.
- Body: **2 to 6 sentences**, plain prose paragraph. No headers, no bullets, no bold sections.
- Emojis allowed where natural, not required.
- Cover what the lead said, what their inquiry is about, and any obvious next-step or urgency cue.

**Implementation:**
- Update the `analyzeCallTranscript` prompt to output JSON like:
  ```json
  {
    "summary": "Plain paragraph hereБ─╕",
    "temperature": "hot" | "warm" | "cold",
    "name": "Б─╕",
    "property_address": "Б─╕",
    "recommended_followup_date": "YYYY-MM-DD",
    "recommended_followup_reason": "Б─╕"
  }
  ```
- Persist `temperature` to a column on `leads` (add column `temperature TEXT` if it doesn't already exist Б─■ check schema; Phase 7C may already have it under another name like `suggested_status`).
- Render summary on the lead card as a single block: emoji badge + label on first line, paragraph below. Replace the existing `ai_summary` display.

**Acceptance:** New calls (and re-analyzed old calls) show the new format. Temperature renders as a colored badge in the AI summary block AND drives the Temperature filter on the Leads tab (see Task 7).

### 5. Simplify lead status Б─■ split lifecycle from temperature

**Problem:** The current status set (`new` / `contacted` / `active` / `hot` / `warm` / `nurture` / `dead`) mixes lifecycle (where the lead is in the funnel) with temperature (how hot the lead is). Result: too many manual choices, leads get lost.

**New model Б─■ two orthogonal axes:**

**Lifecycle (Ryan-controlled, manual click):**
- `new` Б─■ default on intake
- `contacted` Б─■ Ryan made first contact
- `active` Б─■ actively working it
- `dead` Б─■ done, no further follow-up

**Temperature (AI-controlled, auto-set from `analyze-call`):**
- `hot` П÷■╔
- `warm` Б≤─О╦▐
- `cold` Б²└О╦▐ (replaces `nurture`)

Ryan never manually clicks temperature. It's a read-only badge that updates as new activity arrives (existing Phase 7C caching pattern Б─■ regenerate on new activity).

**Action flags stay as-is (orthogonal booleans, Phase 7C):**
- `is_dnc` (red pill)
- `is_junk` (gray pill)
- `is_bad_number` (orange pill)

**Migration:**
- Schema: keep the existing `status` column for lifecycle. Add `temperature TEXT` column if it doesn't exist (check Phase 7C Б─■ may already be there).
- Map existing rows:
  - `hot` Б├▓ status=`active`, temperature=`hot`
  - `warm` Б├▓ status=`active`, temperature=`warm`
  - `nurture` Б├▓ status=`contacted`, temperature=`cold`
  - `new`, `contacted`, `active`, `dead` Б├▓ unchanged (temperature=null until next analyze run)
- Drip engine WHERE clause: continue using lifecycle for hard stops (`dead`, `is_dnc`). Temperature does not gate drip.
- `analyze-call` + `applyAnalyzeCallResult`: write `temperature` field (do NOT overwrite `status` Б─■ lifecycle stays Ryan's).

**UI:**
- Lifecycle dropdown on the lead card: 4 options only.
- Temperature badge: rendered from `temperature` column, color-coded, NOT editable.
- Status pills on the lead card header: render small pills for `is_dnc` / `is_junk` / `is_bad_number` when set, so Ryan can see flag state at a glance without expanding the card.

**Acceptance:** Lifecycle dropdown shows 4 values. Temperature shows as auto-badge driven by AI. DNC / Junk / Bad-# show as colored pills on the collapsed lead card header. Existing 104 leads migrated cleanly.

### 6. Consolidate Leads tab filter chips

**Problem:** Too many filter chips in one row on the Leads tab. Visual noise on mobile.

**New layout Б─■ Gmail-style filter bar:**

**Top row (always visible):**
```
[All] [New] [Contacted] [Active] [Dead]              П÷■█ Filter
```

**Tap П÷■█ Filter Б├▓ secondary filter sheet drops down:**
```
Source:  [All]  [Direct Mail]  [Google Ads]
Temp:    [All]  [П÷■╔ Hot]  [Б≤─О╦▐ Warm]  [Б²└О╦▐ Cold]
Hide:    [ ] DNC   [ ] Junk   [ ] Bad #
```

**Active secondary filters render as removable pills next to the main chips:**
```
[All] [New] [Contacted] [Active] [Dead]   П÷■╔ Hot Б°∙   Direct Mail Б°∙    П÷■█ Filter
```

**Implementation notes:**
- Top row is always one tap to the most-clicked filter (lifecycle).
- Filter sheet is collapsed by default Б─■ saves vertical space on mobile.
- Active filters are visible & one-tap dismissible.
- Default state when first loaded: lifecycle = All, source = All, temp = All, no hides.

**Files likely touched:** `components/widgets/LeadsTab.tsx` (and any related filter state hooks).

**Acceptance:** Leads tab loads with a single clean chip row + Filter button. Setting a secondary filter adds a removable pill. Filter combinations work correctly (lifecycle + source + temp + hides all stack via AND).
